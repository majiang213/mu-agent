## 第一章：项目概览（5分钟读懂）

### 这个项目是什么

local-agent 是一个运行在本地的 ReAct 编程 Agent，专门为 7B/8B 小模型设计（Qwen、DeepSeek，通过 Ollama 运行）。用 TypeScript 写成，基于 pi 框架。

核心目标很明确：让小模型也能可靠地完成编程任务。

### 小模型的五个痛点

大模型（GPT-4、Claude）之所以好用，是因为它们有足够大的上下文窗口、足够强的工具调用能力、足够好的规划能力。小模型全都不够。具体体现在五个痛点：

| 痛点 | 表现 |
|------|------|
| 上下文爆炸 | 7B 模型上下文窗口小，几轮对话就满了 |
| 错误循环 | 卡在同一个错误里反复重试，无法自救 |
| 工具调用弱 | 调用错工具、格式不对、参数瞎填 |
| 靠猜定位代码 | 不知道改哪个文件，随机猜测 |
| 无法处理复杂任务 | 多步骤任务规划能力差 |

### 核心设计哲学：受约束的 ReAct 循环

传统 ReAct Agent 让 LLM 自由决定"下一步做什么"。local-agent 的思路相反：**用确定性管道限制 LLM 的行为边界**。

类比：想象你在教一个刚入职的实习生。你不会说"去把这个 bug 修了"然后任他自由发挥。你会说"第一步，先找到出错的文件；第二步，读懂那段代码；第三步，改；第四步，跑测试验证"。每一步都有明确的任务和工具范围。

这就是 local-agent 的做法。状态机把任务切成若干状态（LOCATE、MODIFY、VERIFY...），每个状态只开放特定的工具，LLM 在这个小盒子里工作，做完了调用 `complete()` 宣告完成，进入下一个状态。

实验数据：确定性管道比 LLM 自主规划的成功率高 40%+。

### 和传统 Agent 框架的本质区别

```
传统框架（LangChain 等）：
用户输入 → LLM 自由规划 → 工具调用 → LLM 决定下一步 → ...

local-agent：
用户输入 → REASON 状态（LLM 制定步骤列表）
         → LOCATE 状态（只能用搜索工具）
         → MODIFY 状态（只能用编辑工具）
         → VERIFY 状态（只能用 bash 验证）
         → DONE
```

关键区别：**工具访问权限由状态机控制，不由 LLM 决定**。LLM 只做它擅长的事：分类、生成、推理。决策逻辑是确定性代码。

## 第三章：核心概念深讲

### State 状态机：15个状态的设计逻辑

```typescript
export enum State {
  LOCATE,       // 找代码位置
  MODIFY,       // 修改代码
  VERIFY,       // 验证修改
  DONE,         // 完成
  REASON,       // 规划步骤
  CLARIFY,      // 向用户澄清需求
  ANSWER,       // 直接回答（不需要改代码）
  DIAGNOSE,     // 诊断 bug 根因
  REVIEW,       // 代码 review
  TEST_WRITE,   // 写测试
  REFACTOR_PLAN,// 规划重构
  ROLLBACK,     // 回滚变更
  RUN,          // 执行命令
  RESEARCH,     // 研究/调查
  SETUP         // 初始化设置
}
```

状态转换规则不是任意的，有明确的路径：

```
REASON → 决定走哪条路径

路径1（普通修改）：LOCATE → MODIFY → VERIFY → DONE
路径2（有 bug）：  DIAGNOSE → LOCATE → MODIFY → VERIFY → DONE
路径3（纯问答）：  ANSWER → DONE
路径4（调研）：    RESEARCH → DONE
路径5（重构）：    REFACTOR_PLAN → LOCATE → MODIFY → VERIFY → DONE
```

为什么要有这么多状态？因为每个状态对应不同的工具集和不同的 system prompt 指令。LOCATE 状态下 LLM 知道"我的任务是找文件"，只有搜索工具可用；MODIFY 状态下 LLM 知道"我的任务是改代码"，只有编辑工具可用。这种约束让小模型专注，不会"想一出是一出"。

### complete() 工具：结构化输出的关键

为什么不直接解析 LLM 的文本输出？因为小模型的输出格式不稳定。让 LLM 调用一个工具，工具参数有 JSON schema 约束，比解析自由文本可靠得多。

`complete()` 是每个状态的"结束信号"。LLM 做完该状态的工作后，必须调用 `complete()` 并传入结构化结果。TypeBox schema 在运行时验证参数格式：

```typescript
// LOCATE 状态的 complete schema
{
  locations: [{
    file: string,      // 必须是字符串
    startLine: number, // 必须是数字
    endLine: number,
    snippet: string    // 代码片段
  }]
}

// VERIFY 状态的 complete schema
{
  passed: boolean,    // 验证是否通过
  issues: string[],   // 发现的问题列表
  summary: string     // 总结
}
```

当 LLM 调用 `complete()` 且参数通过 TypeBox 验证后，`afterToolCall` hook 会调用 `agent.abort()`，终止当前步骤的 pi Agent 循环。这是步骤结束的唯一方式。

### SteerMessage：系统控制信号

SteerMessage 是一种特殊的消息类型，用于系统向 LLM 发送控制信号（比如"你在重复调用同一个工具，请换个思路"）。

为什么不用 `role: 'user'`？因为 user 消息在语义上是"人类说的话"，会被 ContextCompactor 按用户消息的规则处理（截断到300字符）。steer 消息有自己的处理逻辑：在 `transformContext` 中只保留最新一条，旧的全部删除，避免污染上下文。

TypeScript 的 declaration merge 允许扩展第三方类型：

```typescript
// pi-agent-core 原来的类型
interface CustomAgentMessages {
  // 空的，用于扩展
}

// local-agent 里扩展它
declare module '@mariozechner/pi-agent-core' {
  interface CustomAgentMessages {
    steer: SteerMessage;  // 现在 pi Agent 知道有 steer 这种消息类型了
  }
}
```

发给 LLM 之前，`convertToLlm` hook 会把 steer 消息转成 `role: 'user'`，因为 LLM API 只认识 user/assistant/system 这几种角色。

### ExecutionEvent：事件驱动架构

ReactAgent 不直接操控 TUI，而是发出事件。TUI 订阅这些事件并更新界面。好处是执行引擎和 UI 完全解耦，同一个 ReactAgent 可以接不同的 UI（或者不接 UI，用于测试）。

```typescript
export type ExecutionEvent =
  | { type: 'state_change'; from: string; to: string }      // 状态切换
  | { type: 'tool_call'; tool: string; args?: Record<string, unknown> }  // 工具调用
  | { type: 'tool_result'; tool: string; isError: boolean } // 工具结果
  | { type: 'llm_output'; content: string }                 // LLM 输出文本
  | { type: 'llm_thinking'; content: string }               // LLM 思考过程
  | { type: 'llm_call'; promptLen: number; responseLen: number; contextTokens: number }
  | { type: 'task_start'; taskIndex: number; taskTotal: number; description: string }
  | { type: 'clarification_needed'; questions: string[] };
```

## 第四章：执行引擎详解（agent.ts 逐层解析）

agent.ts 是整个项目的心脏，722行。理解它就理解了 local-agent 的工作方式。

### ReactAgent.run() 的完整执行流程

```typescript
// ReactAgent 是无状态的——没有实例字段，每次 run() 都是独立的
export class ReactAgent {
  private _activeAgents: Set<Agent> = new Set();  // 只用于 abort() 支持

  async run(
    input: string,
    modelName: string,
    provider: string,
    baseUrl: string,
    onEvent?: (event: ExecutionEvent) => void,
    initialMessages?: AgentMessage[],
  ): Promise<StateResult> {
    // Step 1: 构建模型对象 + 读取配置
    const model = buildModel(modelName, provider, baseUrl);
    const agentConfig = ConfigManager.getInstance().getConfig();

    // Step 2: 压缩传入的历史对话（独立函数，非实例方法）
    const conversationHistory = compressConversationHistory(
      initialMessages ?? [],
      agentConfig.stateMachine,  // enableCompaction + compactionThreshold
    );

    // Step 3: REASON 状态 - 让 LLM 制定执行计划
    const { steps, needsClarify, questions } =
      await runReasonStep(mission, cfg, conversationHistory, onEvent);
    // steps = [{ state: LOCATE, focus: "找登录相关文件" }, ...]

    // Step 4: 按顺序执行每个步骤（串行，非并行）
    const stepResults: StepHandoff[] = [];
    for (let i = 0; i < steps.length; i++) {
      const handoff = await runStep(steps[i], i, steps.length, mission, stepResults, cfg, onEvent);
      stepResults.push(handoff);
      // 每步的 output 通过 StepHandoff 传递给下一步，不是共享全局历史
    }

    return { state: State.DONE, success: true, output: stepResults.at(-1)?.output ?? '' };
  }
}
```

### buildStepAgent() 的四个 hook

每个步骤都会创建一个新的 pi Agent 实例，配置四个关键 hook：

**1. beforeToolCall - 调用前拦截**

```typescript
beforeToolCall: async (toolCtx) => {
  const toolName = toolCtx.toolCall.name;
  const args = toolCtx.args as Record<string, unknown>;

  // 拦截重复读取：同一个文件已经读过了，不再读
  // readFiles 是 runStep() 里创建的 Set<string>，通过闭包传入
  if (toolName === 'read') {
    const fp = args['filePath'] as string;
    if (readFiles.has(fp)) {
      return { block: true, reason: `Already read: ${fp}. Do not re-read.` };
      //       ^^^^^ 注意：是 block 不是 skip（pi Agent API）
    }
    readFiles.add(fp);
  }

  // 检查文件数量上限（SMALL 模型只能同时处理 2 个文件）
  // 上限追踪在 StateMachineAgent 内部，不是本地变量
  if (toolName === 'edit' || toolName === 'write') {
    if (!cfg.stateMachine.canModifyMoreFiles(cfg.safetyConfig.maxFilesPerTask)) {
      return { block: true, reason: `File modification limit reached (max ${cfg.safetyConfig.maxFilesPerTask})` };
    }
    // 创建检查点（改代码前先备份，出错可 restore）
    if (cfg.safetyConfig.enableCheckpoint) {
      await cfg.safeModifier.createCheckpoint(args['filePath'] as string);
    }
  }
  return undefined;  // 不拦截，正常执行
}
```

**2. afterToolCall - 调用后处理**

```typescript
afterToolCall: async (tool, args, result) => {
  // complete() 成功 → 终止当前步骤
  if (tool === 'complete' && !result.isError) {
    agent.abort();  // 这是步骤结束的唯一触发点
  }

  // 记录编辑过的文件（用于后续更新图索引）
  if (tool === 'edit' || tool === 'write') {
    editedFiles.add(args.path);
  }
}
```

**3. transformContext - 上下文清理**

```typescript
transformContext: (messages) => {
  // 只保留最新的 steer 消息，删除所有旧的
  // 避免上下文里堆积大量"你在循环，换个思路"这类控制信号
  const latestSteer = messages.filter(m => m.role === 'steer').at(-1);
  return messages.filter(m => m.role !== 'steer' || m === latestSteer);
}
```

**4. convertToLlm - 消息格式转换**

```typescript
convertToLlm: (messages) => {
  // LLM API 只认识 user/assistant/system
  // steer 消息在发给 LLM 前转成 user role
  return messages.map(m =>
    m.role === 'steer' ? { ...m, role: 'user' } : m
  );
}
```

### runReasonStep() 的重试逻辑

REASON 步骤是整个执行的规划阶段，有容错机制：

```typescript
// runReasonStep 是模块级函数（非实例方法），重试次数硬编码为最多2次额外调用
async function runReasonStep(mission, cfg, conversationHistory, onEvent) {
  // 第一次运行：LLM 调用 complete(steps=[...]) 提交计划
  await runStepAgent(agent, mission.description, cfg, stagnationDetector,
    () => capturedComplete !== null  // isCompleted 检查
  );

  // 情况1: complete() 根本没有被调用
  if (capturedComplete === null) {
    agent.steer({ role: 'steer', content: '[REMINDER] You must call complete()...', timestamp: Date.now() });
    await runStepAgent(agent, '', cfg, stagnationDetector, () => capturedComplete !== null);
    // 如果这次 complete() 了且 steps 有效 → return steps
    // 否则 → fallback
  }

  // 情况2: complete() 被调用了，但 steps 数组无效
  if (capturedComplete !== null) {
    const { steps, error } = parseReasonSteps(capturedComplete);
    if (steps.length > 0) return { steps, needsClarify: false, questions: [] };

    // steps 无效：steer 告知具体错误，再重试一次
    capturedComplete = null;
    agent.steer({ role: 'steer',
      content: `[ERROR] complete() was called but plan is invalid. ${error} Fix and call complete() again.`,
      timestamp: Date.now()
    });
    await runStepAgent(agent, '', cfg, stagnationDetector, () => capturedComplete !== null);
    if (capturedComplete !== null) {
      const { steps: retrySteps } = parseReasonSteps(capturedComplete);
      if (retrySteps.length > 0) return { steps: retrySteps, needsClarify: false, questions: [] };
    }
  }

  // 最终回退：默认步骤序列（不依赖 modelParams.maxRetries 配置）
  return { steps: DEFAULT_FALLBACK_STEPS(mission.description) };
  // DEFAULT_FALLBACK_STEPS = [LOCATE, MODIFY, VERIFY]
}
```

### runStep() 的 complete 等待机制

每个步骤的执行本质上是：启动 pi Agent，等待 LLM 调用 `complete()`，然后返回结果。

```typescript
async runStep(step, task, history, modelParams): Promise<StepResult> {
  // 1. 代码定位（LOCATE/DIAGNOSE/RESEARCH 等状态）
  if (STATES_NEEDING_LOCATOR.includes(step.state)) {
    const locatorResult = await this.locator.locate(step.focus);
    // 结果会注入到 system prompt 的 <project_structure> 和 <suggested_files> 块
  }

  // 2. 构建这个状态专用的 complete 工具
  let completeResult: unknown = null;
  const completeTool = buildCompleteTool(step.state, (result) => {
    completeResult = result;  // complete() 被调用时保存结果
  });

  // 3. 创建 pi Agent，只开放该状态允许的工具 + complete 工具
  const agent = buildStepAgent(step, completeTool, ...);

  // 4. 运行 pi Agent（内部是 LLM ↔ 工具 的循环）
  await agent.run();

  // 5. MODIFY 完成后，更新代码图索引
  if (step.state === State.MODIFY && editedFiles.size > 0) {
    await this.graphBuilder.updateFiles([...editedFiles]);
  }

  return { status: completeResult ? 'done' : 'failed', data: completeResult };
}
```

## 第五章：智能防护机制

小模型最容易出的问题就是"卡住"：反复调用同一个工具、在同一个错误里打转、或者什么有效的事都不做。local-agent 有一套防护机制来检测和处理这些情况。

### StagnationDetector：4种检测算法

**算法1：重复工具调用检测**

```typescript
detectRepeatedToolCalls(calls: ToolCall[], threshold = 3): boolean {
  // 检查最近 N 次调用是否完全相同（工具名 + 参数）
  if (calls.length < threshold) return false;
  const last = calls.slice(-threshold);
  return last.every(c => c.tool === last[0].tool &&
    JSON.stringify(c.args) === JSON.stringify(last[0].args));
}
// 例：连续3次 read("src/utils.ts") → 触发
```

**算法2：重复错误检测**

```typescript
detectRepeatedErrors(errors: string[], threshold = 2): boolean {
  // 同一个错误出现 N 次
  if (errors.length < threshold) return false;
  const last = errors.slice(-threshold);
  return last.every(e => e === last[0]);
}
// 例：连续2次 "File not found: src/auth.ts" → 触发
```

**算法3：Cycle 检测（最复杂）**

```typescript
detectCycle(calls: ToolCall[], windowSize = 3): boolean {
  // 滑动窗口：检查最近 W 次调用是否在重复之前 W 次的模式
  // 例：[read, grep, read, grep] → 窗口W=2，后半段 [read,grep] == 前半段 [read,grep] → 触发
  if (calls.length < windowSize * 2) return false;
  const recent = calls.slice(-windowSize);
  const before = calls.slice(-windowSize * 2, -windowSize);
  return recent.every((c, i) =>
    c.tool === before[i].tool &&
    JSON.stringify(c.args) === JSON.stringify(before[i].args)
  );
}
```

**算法4：无进展检测**

```typescript
detectNoProgress(calls: ToolCall[], threshold = 5): boolean {
  // 连续 N 次都是只读操作（read/grep/find/ls），没有任何写操作
  if (calls.length < threshold) return false;
  const last = calls.slice(-threshold);
  const readOnlyTools = new Set(['read', 'grep', 'find', 'ls']);
  return last.every(c => readOnlyTools.has(c.tool));
}
// 注意：READ_ONLY_STATES（LOCATE、RESEARCH 等）禁用此检测
```

### 两级升级策略

```
第一次检测到停滞：
  → 发送 steer 消息（"你在重复调用同一工具，请换个思路"）
  → 重置检测器（给 LLM 一次机会）

第二次检测到停滞：
  → agent.abort()（强制终止当前步骤）
  → 步骤标记为 failed
```

### ContextCompactor：结构化压缩

ContextCompactor 在两个地方运行，解决两个不同的问题：

**1. 跨轮次压缩（run() 入口）**

每次 `ReactAgent.run()` 开始时，对传入的历史消息做一次压缩，防止多轮对话积累的历史撑爆 context：

```typescript
const conversationHistory = compressConversationHistory(initialMessages ?? [], cfg.smConfig);
```

**2. 循环内压缩（transformContext hook）**

每次 LLM 调用前，`transformContext` 都会运行，在清理旧 steer 消息之后，检查当前消息列表是否超过 `IN_LOOP_TOKEN_BUDGET = 24000` tokens（32k context 的 75%）。超过则自动压缩：

```typescript
transformContext: async (messages) => {
  // Step 1: 只保留最新的 steer 消息
  const latestSteer = steerIndices.at(-1);
  let result = messages.filter((m, i) => m.role !== 'steer' || i === latestSteer);

  // Step 2: 超过 token 预算则压缩中间消息
  const compactor = new ContextCompactor({ maxTokens: IN_LOOP_TOKEN_BUDGET });
  return compactor.compact(result).messages;
}
```

这解决了单步骤内（如 LOCATE 读取大量文件）context 溢出的问题。

**压缩规则（两处相同）：**

```
原始消息列表：[msg1, msg2, ..., msg50]

压缩后：
  head: [msg1, msg2]        ← 前2条保留完整（初始任务描述）
  tail: [msg45...msg50]     ← 最后6条保留完整（最新上下文）
  middle: [压缩版 msg3...msg44]

middle 压缩规则：
  steer 消息    → 完全删除（控制信号不需要保留历史）
  user 消息     → 截断到前300字符
  assistant 消息 → tool_calls 替换为 "[tool:read]"，文本截断到200字符
  toolResult 消息 → 替换为 "read✓" 或 "read✗"（成功/失败标记）
```

不用 LLM 摘要：LLM 摘要本身消耗 token，且可能摘掉关键信息。结构化提取可预测、速度快、不出错。

### 重复读拦截

在 `buildStepAgent` 的 `beforeToolCall` hook 里维护一个 `readFiles: Set<string>`。同一步骤内，同一个文件只读一次。LLM 有时会反复读同一个文件（可能是"确认"行为），这个拦截节省了大量无效 token。

## 第六章：代码定位系统（BM25 + 调用图）

### 为什么需要专门的代码定位

让小模型自己找代码位置，它会随机猜。"修复登录 bug" → 它可能去改 `src/utils.ts`，而实际的登录逻辑在 `src/auth/service.ts`。

`needsRebuild()` 检查 `git HEAD` commit hash 是否变化，只有代码变更后才重建图。`updateFiles(paths)` 支持增量更新，MODIFY 步骤完成后只更新被编辑的文件。

### BM25 算法 + 2-hop 图扩展

BM25 是一种经典的文本相关性算法（搜索引擎常用），比简单的关键词匹配更准确：

```
查询："login authentication"
候选函数：
  - handleLogin() in auth/service.ts    → BM25 分数: 0.85  ← 高分
  - validateToken() in auth/jwt.ts      → BM25 分数: 0.72
  - updateUserProfile() in user/api.ts  → BM25 分数: 0.12
```

2-hop 图扩展：BM25 找到 `handleLogin()` 后，再沿调用图扩展一跳，把 `handleLogin()` 调用的函数也加进来：

```
handleLogin() → calls → validatePassword(), createSession(), logEvent()
                           ↑ 这些也加入结果集
```

这样即使 `validatePassword()` 的名字和查询不相关，也能被找到。

```sql

  file TEXT,          -- 文件路径
  start_line INTEGER,
  end_line INTEGER,
  snippet TEXT        -- 代码片段（前几行）

-- 边表：调用关系

  from_node INTEGER REFERENCES nodes(id),
  to_node INTEGER REFERENCES nodes(id)
);
```

### buildSystemPrompt() 的组装逻辑

system prompt 不是一个固定字符串，而是动态组装的：

```typescript
function buildSystemPrompt(state: State, step: Step, modelParams: ModelParams, locatorResult?: LocatorResult): string {
  const parts = [
    BASE_PROMPT,                              // 基础行为规则 + 环境信息
    STATE_INSTRUCTIONS[state],               // 当前状态的具体指令
    modelParams.tier === 'SMALL'
      ? SMALL_MODEL_CONSTRAINTS              // 小模型额外约束
      : '',

      ? buildLocatorSection(locatorResult)   // 代码结构和建议文件
      : ''
  ];

}
```

## 附录：设计决策记录

### 为什么用 pi 框架而不是 LangChain

LangChain 是为大模型设计的，抽象层太多，调试困难，且假设 LLM 有强大的规划能力。pi 框架更轻量，暴露底层的 `Agent` 类和 hook 系统，方便 local-agent 在工具调用前后插入自己的逻辑（beforeToolCall、afterToolCall）。pi 的 `streamSimple` 兼容 OpenAI API 格式，Ollama 也支持这个格式，天然适配。

### 为什么 complete() 而不是 JSON 解析

直接解析 LLM 输出的 JSON 有几个问题：
1. 小模型经常输出格式不规范的 JSON（多余的注释、尾随逗号等）
2. 解析失败时不知道该如何重试（没有结构化的错误信息）
3. LLM 可能在 JSON 前后加大量解释文字，很难提取

`complete()` 工具调用的参数是严格的 JSON，LLM 在调用工具时会更认真地格式化参数。加上 TypeBox 的运行时验证，格式错误时可以给出具体的错误信息（"steps[0].state 必须是有效的 State 枚举值"），让重试逻辑更精准。

### 为什么 SteerMessage 不用 role:user

如果把 steer 消息放进 `role: 'user'`，有两个问题：

1. **语义混淆**：ContextCompactor 会按用户消息规则处理（截断到300字符），但 steer 消息通常比较短，截断不是问题；更大的问题是 LLM 可能把 steer 消息当成用户的新需求，改变任务目标。

2. **清理困难**：用 steer 这个自定义 role，`transformContext` 可以精确识别并只保留最新一条，旧的全删。如果混在 user 消息里，很难区分哪些是控制信号、哪些是真正的用户输入。

### 为什么 ContextCompactor 不用 LLM 摘要

LLM 摘要有几个实际问题：

1. **成本**：摘要本身需要一次 LLM 调用，消耗 token。在上下文已经很长的情况下，摘要调用可能本身就很贵。

2. **延迟**：多一次 LLM 调用就多几秒等待时间。

3. **不可预测**：LLM 摘要可能丢失关键信息（比如某个文件的具体行号）。结构化提取虽然粗糙，但丢失的信息是可预测的（只是截断，不是语义压缩）。

4. **递归问题**：如果摘要 LLM 本身也有上下文限制，可能需要摘要的摘要，变得复杂。

### 为什么删除 maxIterations

早期版本有一个 `maxIterations` 参数限制整个任务的总迭代次数。删除的原因：

1. **粒度太粗**：不同任务的合理迭代次数差异很大，一个全局上限很难设置合适。

2. **被更精细的机制取代**：现在有 `StagnationDetector`（检测真正的停滞）+ `ContextCompactor` 的 in-loop 压缩（防止 context 溢出）。这两个机制比一个全局计数器更智能。

3. **用户体验**：复杂任务因为 maxIterations 被截断，用户体验很差。现在只在真正检测到问题时才中断。
